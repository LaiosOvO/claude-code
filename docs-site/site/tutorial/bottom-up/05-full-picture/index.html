    5.2 关键设计决策回顾¶

    为什么用 AsyncGenerator？¶

    对比方案：
  回调函数  → 回调地狱，难以组合
  Promise   → 无法流式，必须等全部完成
  Observable → 需要 RxJS，学习成本高
  AsyncGen  → 原生支持，流式，可组合，可背压 ✓

    为什么用 React 写终端 UI？¶

    对比方案：
  console.log  → 无法更新已输出的内容
  ncurses      → C 绑定，不支持组件模型
  blessed      → 过时，不维护
  React + Ink  → 声明式，组件化，生态成熟 ✓

    为什么安全属性默认 false？¶

    Fail-Closed vs Fail-Open：
  Fail-Open:   忘了标记 → 默认允许 → 可能执行危险操作 ✗
  Fail-Closed: 忘了标记 → 默认拒绝 → 最多是功能降级 ✓

  安全系统永远选择 Fail-Closed

    为什么工具要分三个文件？¶

    变化频率不同：
  执行逻辑  → 功能稳定后很少改
  模型提示  → 优化提示词会频繁调整
  UI 渲染   → 界面经常调整

分开文件让不同关注点独立迭代

    5.4 从 Claude Code 学到的架构模式¶

    模式 1：快速路径优先¶

    先处理简单情况，最后才走复杂路径。减少不必要的模块加载。

    模式 2：安全优先的默认值¶

    所有可选标记的默认值选择最安全的选项。宁可功能降级，不冒安全风险。

    模式 3：三层关注点分离¶

    逻辑 / 提示词 / UI 各自独立。不同的受众（引擎/AI/用户）各有所需。

    模式 4：AsyncGenerator 驱动¶

    整个查询循环用异步生成器驱动，天然支持流式和背压。

    模式 5：传输抽象¶

    不管底层是 WebSocket、SSE 还是 HTTP，上层用统一接口。传输方式可以切换而不影响业务逻辑。

    模式 6：Feature Gate（特性开关）¶

    用 feature('FLAG') 控制功能启停。构建时可以死代码消除。

    模式 7：文件级惰性加载¶

    require() / import() 动态加载打破循环依赖。代码在需要时才加载。

    模式 8：有界数据结构¶

    BoundedUUIDSet、LRUCache 等有界集合防止内存泄漏。

    5.5 学习路径建议¶

    如果你想**修改或扩展**这个项目，建议的切入路径：

    初级：理解启动流程
  → 读 entrypoints/cli.tsx → main.tsx → setup.ts → replLauncher.tsx
  → 大约 2 小时

中级：理解对话引擎
  → 读 query.ts → QueryEngine.ts → Tool.ts
  → 试着添加一个简单的 Tool
  → 大约 4 小时

高级：理解远程桥接
  → 读 bridge/ 目录的核心文件
  → 理解 WebSocket transport
  → 大约 6 小时

专家：理解完整 REPL
  → 读 screens/REPL.tsx (5000行)
  → 理解 hooks/ 的组合模式
  → 大约 8 小时

    5.7 结语¶

    你现在已经从最底层的 Signal 原语，一路理解到了完整的系统架构。

    这个项目的核心就是一句话：

    用 React 在终端中构建 UI，用 AsyncGenerator 驱动 AI Agent 循环，用 Tool 系统赋予 AI 执行能力，用 Bridge 实现远程协作。

    一切复杂性都围绕这个核心展开。当你遇到不理解的代码时，问自己：它在为这四件事中的哪一件服务？

    祝你学习愉快！
