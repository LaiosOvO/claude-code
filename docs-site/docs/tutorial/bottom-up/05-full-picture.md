## 5.2 关键设计决策回顾

### 为什么用 AsyncGenerator？
```
对比方案：
  回调函数  → 回调地狱，难以组合
  Promise   → 无法流式，必须等全部完成
  Observable → 需要 RxJS，学习成本高
  AsyncGen  → 原生支持，流式，可组合，可背压 ✓
```

### 为什么用 React 写终端 UI？
```
对比方案：
  console.log  → 无法更新已输出的内容
  ncurses      → C 绑定，不支持组件模型
  blessed      → 过时，不维护
  React + Ink  → 声明式，组件化，生态成熟 ✓
```

### 为什么安全属性默认 false？
```
Fail-Closed vs Fail-Open：
  Fail-Open:   忘了标记 → 默认允许 → 可能执行危险操作 ✗
  Fail-Closed: 忘了标记 → 默认拒绝 → 最多是功能降级 ✓
  
  安全系统永远选择 Fail-Closed
```

### 为什么工具要分三个文件？
```
变化频率不同：
  执行逻辑  → 功能稳定后很少改
  模型提示  → 优化提示词会频繁调整
  UI 渲染   → 界面经常调整

分开文件让不同关注点独立迭代
```

## 5.4 从 Claude Code 学到的架构模式

### 模式 1：快速路径优先
先处理简单情况，最后才走复杂路径。减少不必要的模块加载。

### 模式 2：安全优先的默认值
所有可选标记的默认值选择最安全的选项。宁可功能降级，不冒安全风险。

### 模式 3：三层关注点分离
逻辑 / 提示词 / UI 各自独立。不同的受众（引擎/AI/用户）各有所需。

### 模式 4：AsyncGenerator 驱动
整个查询循环用异步生成器驱动，天然支持流式和背压。

### 模式 5：传输抽象
不管底层是 WebSocket、SSE 还是 HTTP，上层用统一接口。传输方式可以切换而不影响业务逻辑。

### 模式 6：Feature Gate（特性开关）
用 `feature('FLAG')` 控制功能启停。构建时可以死代码消除。

### 模式 7：文件级惰性加载
`require()` / `import()` 动态加载打破循环依赖。代码在需要时才加载。

### 模式 8：有界数据结构
`BoundedUUIDSet`、`LRUCache` 等有界集合防止内存泄漏。
